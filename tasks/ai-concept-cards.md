## 1. LLM（大语言模型）

**一句话解释：**
LLM 是一个通过海量文本训练出来的"概率接词器"——给它一段话，它预测下一个最可能的 token，反复迭代就生成了回答。

**具体例子：**
你问 ChatGPT "以太坊的共识机制是"，它不是去查数据库，而是根据训练数据中见过的模式，一个字一个字地"接"出 "Proof of Stake"。本质上和手机输入法的联想功能是同一个原理，只是规模大了几万倍。

**常见误解：**
- ❌ "LLM 理解了你说的话" — 它不理解，它只是统计上最可能的输出
- ❌ "LLM 知道所有最新信息" — 训练数据有截止日期，过了那个时间的事它不知道
- ❌ "模型越大越准" — 参数量只是因素之一，数据质量和对齐训练（RLHF）同样关键

---

## 2. Prompt（提示词）

**一句话解释：**
Prompt 就是你给 LLM 的"指令文本"，它决定了模型输出的方向、格式和质量。

**具体例子：**
同一个问题，不同 prompt 效果天差地别：
- 差的 prompt："写个智能合约"
- 好的 prompt："用 Solidity 写一个 ERC-20 代币合约，包含 mint 和 burn 功能，添加 NatSpec 注释，考虑重入攻击防护"

后者输出的代码可以直接用，前者可能给你一个漏洞百出的 Hello World。

**常见误解：**
- ❌ "Prompt 工程是玄学" — 其实是有规律的：角色设定 + 任务描述 + 输出格式 + 约束条件
- ❌ "Prompt 越长越好" — 关键是信息密度，废话会稀释重点
- ❌ "写好 prompt 就不需要理解技术了" — 你得懂业务才能写出精准的 prompt

## 3. RAG（检索增强生成）

**一句话解释：**
RAG = 先从外部知识库检索相关内容，再把检索结果塞进 prompt 让 LLM 基于这些内容回答，解决 LLM "不知道"和"瞎编"的问题。

**具体例子：**
你问 AI Agent："我钱包里的 USDC 余额是多少？" LLM 本身不知道，但 RAG 流程会：
1. 检索：调用链上 API 查到你的余额是 150 USDC
2. 增强：把 "用户余额：150 USDC" 塞进 prompt
3. 生成：LLM 基于真实数据回答 "你当前有 150 USDC"

**常见误解：**
- ❌ "RAG 就是给 LLM 加了个搜索引擎" — 检索质量（chunk 切分、embedding 模型、相似度算法）直接决定回答质量
- ❌ "RAG 能解决所有幻觉问题" — 如果检索到的内容本身有错，LLM 会基于错误内容自信地胡说
- ❌ "RAG 不需要 fine-tuning" — 复杂场景下，RAG + 微调的组合效果更好

## 4. Agent（智能体）

**一句话解释：**
Agent = LLM + 记忆 + 工具 + 规划能力，能自主拆解任务、调用外部工具、根据结果决定下一步，而不只是被动回答问题。

**具体例子：**
你说："帮我查一下 Uniswap 上 ETH/USDC 的价格，如果低于 2000 就在测试网下一个限价单。"

Agent 的执行过程：
1. **规划**：拆成"查价格"和"条件下单"两步
2. **调用工具**：用 Web3 工具查链上价格
3. **判断**：价格是 2100，高于 2000
4. **回复**："当前 ETH 价格 $2100，高于你的目标价 $2000，暂不执行。需要我设置价格提醒吗？"

这就是和普通 ChatBot 的区别——它能**做事**，不只是**说话**。

**常见误解：**
- ❌ "Agent 就是套了个循环的 ChatGPT" — 核心难点在于工具选择的准确性和错误恢复
- ❌ "Agent 能完全自主决策" — 高风险操作（转账、合约调用）必须有人类确认环节（human-in-the-loop）
- ❌ "Agent 很贵很慢" — 取决于任务复杂度，简单工具调用可以很快很便宜

## 5. MCP（Model Context Protocol）

**一句话解释：**
MCP 是一个开放协议，让 AI Agent 能以标准化方式连接各种外部工具和数据源，就像 USB 让电脑统一连接各种外设。

**具体例子：**
没有 MCP 之前，你想让 Agent 同时查 GitHub、读 Notion、调 Etherscan，每个都要写专门的集成代码。有了 MCP：
- GitHub 提供一个 MCP Server
- Notion 提供一个 MCP Server
- Etherscan 提供一个 MCP Server

Agent 只要支持 MCP 协议，就能即插即用地使用所有这些工具，不用为每个工具写适配代码。

**常见误解：**
- ❌ "MCP 是 Anthropic 的私有协议" — 虽然由 Anthropic 提出，但它是开放标准，任何模型/工具都能接入
- ❌ "MCP = Function Calling" — Function Calling 是模型层面的能力，MCP 是连接层面的协议，解决的是不同问题
- ❌ "有了 MCP 就不需要 API 了" — MCP 是 API 的"翻译层"，底层还是在调各种 API

---

## 6. Fine-tuning（微调）

**一句话解释：**
Fine-tuning 是在预训练好的大模型基础上，用特定领域的数据继续训练，让模型在某个任务上表现更好，就像给一个通才做了专项培训。

**具体例子：**
通用 LLM 写 Solidity 代码经常犯错（比如忘记加 nonReentrant 修饰符）。你用 1000 个经过审计的高质量 Solidity 合约作为训练数据做 fine-tuning，模型就学会了：
- 自动加 ReentrancyGuard
- 用 SafeERC20 处理代币转账
- 按照最佳实践组织合约结构

**常见误解：**
- ❌ "Fine-tuning 能让模型学到新知识" — 它更多是调整模型的"行为模式"，而不是注入新事实。注入新知识用 RAG 更合适
- ❌ "Fine-tuning 一定比 prompt engineering 好" — 很多场景下，写好 prompt 就够了，fine-tuning 成本高且有过拟合风险
- ❌ "Fine-tuning 需要海量数据" — 高质量的几百条数据可能比低质量的几万条更有效（数据质量 > 数据数量）

## 总结：它们之间的关系

```
                    ┌─────────────┐
                    │  Fine-tuning │ ← 训练阶段优化模型
                    └──────┬──────┘
                           ▼
┌──────┐    ┌─────┐    ┌──────┐    ┌───────┐
│Prompt │───▶│ LLM │◀───│ RAG  │    │ Agent │
└──────┘    └─────┘    └──────┘    └───┬───┘
                                       │
                                  ┌────▼────┐
                                  │   MCP   │ ← 连接外部工具的标准协议
                                  └─────────┘
```

- **Prompt** 是你和 LLM 对话的接口
- **LLM** 是核心引擎
- **RAG** 让 LLM 能访问外部知识
- **Fine-tuning** 让 LLM 更擅长特定任务
- **Agent** 把以上所有组合起来，能自主完成复杂任务
- **MCP** 让 Agent 能标准化地连接各种工具
